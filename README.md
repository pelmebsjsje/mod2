# LiteBuilder — скелет мода для автопостройки по схематикам Litematica

## Что это
Fabric-мод для Minecraft 1.21.8. Идея: бот читает активную схему,
загруженную в **Litematica**, и реалистично строит её — ходит
(через **Baritone**), берёт блоки из инвентаря и ставит их по одному,
как игрок.

## Что уже готово
- Структура Gradle-проекта под Fabric Loom.
- `LiteBuilderClient` — команды `/litebuild start|pause|stop|status`.
- `SchematicScanner` — обход блоков активного размещения схемы (нужна
  сверка с актуальным API Litematica под 1.21.8).
- `BuildManager` — цикл: очередь → проверка инвентаря → Baritone
  подводит игрока → попытка установки блока.

## Что нужно доделать (важно!)
1. **Точные сигнатуры Litematica API.** Я писал `SchematicScanner` по
   памяти о структуре мода — методы вроде `forEachBlockInRegion` могут
   называться иначе в актуальной версии. Открой исходники Litematica
   на GitHub (ветка под 1.21.8) и сверь:
   - как получить активный `SchematicPlacement`,
   - как итерироваться по блокам региона,
   - как получить трансформацию локальных координат в мировые (поворот/зеркало).

2. **Реальная установка блока (`tryPlaceBlock`).** Нужно:
   - Выбрать нужный предмет в хотбаре (`PlayerInventory.selectedSlot`
     или `Inventory.selectSlotInHotbar`).
   - Сделать raycast от глаз игрока до грани соседнего блока рядом с
     целевой позицией, получить `BlockHitResult`.
   - Симулировать взгляд игрока в нужную сторону (`Entity.setYaw/setPitch`
     или через `LookHelper`/Baritone `RotationUtils` — в Baritone есть
     готовые утилиты для плавного поворота камеры, посмотри `RotationUtils`
     и `IPlayerContext`).
   - Вызвать `client.interactionManager.interactBlock(player, Hand.MAIN_HAND, hitResult)`.

3. **Инвентарь.** Сейчас при отсутствии блока задача просто помечается
   проваленной. Стоит добавить:
   - поиск ближайшего сундука с нужным блоком (через Baritone `GoalNear`
     до позиции сундука + открытие/перекладывание предметов),
   - или крафт на месте, если есть верстак и материалы.

4. **Актуализировать версии в `gradle.properties`.**
   Проверь на:
   - https://fabricmc.net/develop — loader/fabric-api/yarn под 1.21.8,
   - Modrinth/CurseForge страницы MaLiLib и Litematica — версии под 1.21.8,
   - репозиторий `cabaletta/baritone` на GitHub — актуальные теги под 1.21.8
     (Baritone не всегда сразу выходит под новую версию MC — проверь,
     что форк/ветка для 1.21.8 вообще существует на момент сборки).

5. **Безопасность выполнения.** Раз это твой сервер/одиночная игра —
   ограничений со стороны анти-чита нет, но всё равно разумно добавить:
   - лимит скорости построения (не чаще N блоков/сек), чтобы не забивать
     тикующий поток и не выглядеть подозрительно на будущее, если решишь
     использовать на чужом сервере,
   - обработку "блок уже стоит другой" (переставленный игроком) —
     пропуск с логом, а не бесконечный ретрай.

## Сборка
```
./gradlew build
```
Собранный джар — в `build/libs/`. Поставь его вместе с Fabric API,
MaLiLib, Litematica и Baritone в `mods/`.

## Использование
1. Загрузи и активируй схему в Litematica как обычно (превью в мире).
2. `/litebuild start` — бот начнёт строить снизу вверх.
3. `/litebuild status` — прогресс.
4. `/litebuild pause` / `/litebuild stop` — остановка.

## Команды на русском в чат (через #)
Сообщение, начинающееся с `#`, перехватывается модом и НЕ уходит на сервер.

Примеры:
```
#найди тростник в сундуках в радиусе 40 блоков
#копай генератор булыжника блоки на координатах x:100 y:64 z:-32
#иди x:10 y:70 z:5
#строй
#паузу
#остановись
```

Сначала команду пытается разобрать быстрый regex-парсер (`RegexCommandParser`,
без сети, мгновенно). Если формулировка нестандартная — уходит запрос к LLM
(`LlmCommandParser`), которая переводит любую фразу в тот же строгий JSON-формат.
**Важно:** LLM не даёт боту новых способностей — она только гибко сопоставляет
свободный текст с уже реализованными действиями (`find_in_chests`, `mine_area`,
`goto`, `build_start`, `pause`, `stop`). Если попросить что-то, чего в
`CommandAction` нет — вернётся `unknown`.

### Настройка LLM-фолбэка
Файл `config/litebuilder/config.json` создаётся при первом запуске:
```json
{
  "baseUrl": "http://localhost:11434/v1/chat/completions",
  "apiKey": "",
  "model": "llama3.1:8b",
  "llmFallbackEnabled": true
}
```
- **Ollama (бесплатно, без лимитов, локально):** поставь Ollama, `ollama pull llama3.1:8b`,
  оставь `baseUrl` как есть, `apiKey` пустой.
- **Groq (бесплатный тариф с лимитом запросов/мин):** `baseUrl` →
  `https://api.groq.com/openai/v1/chat/completions`, впиши свой ключ, `model` →
  например `llama-3.1-8b-instant`.
- Если совсем не хочешь возиться с LLM — поставь `"llmFallbackEnabled": false`,
  тогда будут работать только команды, которые понимает regex-парсер (их легко
  добавлять самому в `RegexCommandParser`).

### Расширение словаря
`assets/litebuilder/ru_items.json` — русское название → Minecraft ID.
Добавляй новые слова туда без пересборки логики парсера.

## Плавное движение (не как "робот" Baritone)
`MovementSmoother` перехватывает управление камерой: вместо мгновенного snap
на целевой yaw/pitch (как по умолчанию делает Baritone) он каждый тик плавно
доворачивает камеру с ограничением угловой скорости и лёгким шумом.

**Обязательно сделай на своей стороне:**
1. В настройках Baritone (`baritone.api.Settings`) отключи его собственное
   мгновенное управление вращением/взглядом — иначе `MovementSmoother` будет
   конкурировать с Baritone за контроль камеры и получится дёрганое движение.
   Ищи в `Settings` что-то вроде `freeLook` / `blockFreeLook` /
   `antiCheatCompatibility` — конкретные имена и поведение проверь в исходниках
   Baritone под свою версию, они менялись между релизами.
2. Само перемещение (WASD, спринт, прыжки) Baritone продолжает делать сам —
   он и так двигается по сетке путей плавно во времени, резкость именно в
   повороте камеры. Если после отключения ротации Baritone движение всё ещё
   выглядит рывками — добавь аналогичный сглаживатель для ускорения/торможения
   через `player.input` вместо прямого управления скоростью.
3. `tryPlaceBlock` / `tryBreakBlock` сейчас ждут, пока `smoother.isAimedAt(...)`
   покажет, что камера "по-настоящему" смотрит на цель (а не как раньше —
   мгновенно), поэтому взаимодействие с блоком естественно откладывается на
   несколько тиков, пока идёт довод камеры — это тоже часть "человечности".

## Что было сделано в этой доработке (актуально на момент правки)

**Проверено дизассемблированием присланных jar (не по памяти/документации):**
- Litematica 0.23.6, MaLiLib 0.25.7, Baritone API 1.15.0 — все под MC 1.21.6–1.21.8, совместимы.
- Реальные сигнатуры `DataManager`, `SchematicPlacementManager`, `SchematicPlacement`,
  `SchematicWorldHandler`, `IBaritone`, `IBuilderProcess`, `IMineProcess`, `ICustomGoalProcess`,
  `Settings` — см. комментарии в `bridge/BaritoneBridge.java` и `bridge/LitematicaBridge.java`.

**Новое:**
- `bridge/` — единственный слой, трогающий классы Baritone/Litematica/vanilla напрямую.
  `EmergencyStop`, `BaritoneBridge`, `LitematicaBridge`, `VanillaInteractionBridge`.
- `tools/ToolRegistry.java` — все 15 инструментов из ТЗ (goto, build, mine, placeBlock,
  breakBlock, findChest, takeItem, craft, smelt, scanArea, lookAt, openContainer, useItem,
  checkInventory, checkWorld) как `ToolTask` с реальными вызовами API.
- `ai/GroqClient.java` — асинхронный (не блокирующий клиент игры) запрос плана у Groq,
  ключ НЕ хардкодится, берётся из `config/litebuilder/config.json` или переменной окружения
  `GROQ_API_KEY`.
- `ai/AgentMemory.java` — `agent_memory.json`, продолжение задачи после перезапуска клиента.
- `ai/AiAgentManager.java` — исполнитель плана, статусы в чат ("Принял"/"Завершил"/
  "Нужны ресурсы"/"Ошибка"), логи решений в консоль.
- `SchematicScanner` переписан на реальный API (виртуальный мир схемы вместо гипотетических
  методов обхода регионов).
- `BuildManager`/`TaskManager` — реальная установка/разрушение блоков через
  `VanillaInteractionBridge` вместо заглушек, anti-cheat пауза между блоками,
  `EmergencyStop` проверяется каждый тик.
- Команда `#ии <задача>` — свободная постановка задачи ИИ. `#стоп` — мгновенная остановка
  BuildManager + TaskManager + AiAgentManager + Baritone. Любое ОБЫЧНОЕ сообщение в чат
  (не через #) тоже мгновенно замораживает агента (см. `ChatCommandInterceptor`).

**Честные ограничения (не доведено до продакшена в рамках этой правки):**
- `craft`/`smelt` — упрощённая реализация (см. комментарии в `ToolRegistry`): работает для
  типовых shaped/shapeless-рецептов при уже открытом верстаке/печи, не разруливает сложные
  случаи (несколько вариантов ингредиента, автоматический подбор рецепта по приоритету).
- Полный проход `gradle build` **не выполнялся** — в песочнице нет сетевого доступа, чтобы
  Loom мог скачать Minecraft/Yarn/Fabric Loader. Все использованные сигнатуры сверены вручную
  дизассемблированием jar, но финальную сборку и рантайм-тест нужно сделать локально:
  `./gradlew build`, затем проверить `mods/` (baritone jar нужно положить туда физически,
  Gradle его не бандлит — см. комментарий в build.gradle).
- Груз-ключ, который был прислан в задании, засветился в этом чате — отзови его и впиши новый
  в `config/litebuilder/config.json` (`apiKey`) либо через `GROQ_API_KEY`.
